## Correctness Review

### C-1: `--tests` and `--output` argument parsing crashes silently when flag is last argument

**File:** `test-review-request.sh:59-67`
**Severity:** IMPORTANT

**Issue:** When `--tests` or `--output` is the last argument (no value follows), `shift` on line 62/66 will shift with `$#` at 0. With `set -e`, the `${1:-}` expansion prevents unbound variable errors, but the second `shift` on an empty argument list causes a non-zero exit (shift past end), which under `set -e` terminates the script silently. The error message is unhelpful — the user just sees the script exit.

**Fix:** Guard the second shift: `[[ $# -gt 0 ]] && shift` or validate that an argument follows the flag before shifting.

---

### C-2: `--quick` without a following positional argument and no later positional leaves `MODULE_PATH` empty

**File:** `test-review-request.sh:51-57`
**Severity:** MINOR

**Issue:** If invoked as `test-review-request.sh --quick --output ./out`, the `--quick` handler checks `$1` but it matches `^--` so `MODULE_PATH` stays empty. This is caught later at line 80, but the error message says "Module path is required" without hinting that `--quick` expects a positional argument.

**Fix:** The existing guard at line 80 handles this; no code change needed, but the usage message could be clearer.

---

### C-3: `REPO_ROOT` is computed but never used

**File:** `test-review-request.sh:30-33`
**Severity:** NIT

**Issue:** `REPO_ROOT` is assigned (and the script exits if it fails) but the variable is never referenced. This is dead code that adds a git dependency that may not be needed.

**Fix:** Remove the `REPO_ROOT` block, or use it (e.g., for resolving relative paths).

---

### C-4: Quick mode test discovery override order

**File:** `test-review-request.sh:112-114`
**Severity:** MINOR

**Issue:** The quick mode override `TEST_PATH="$MODULE_PATH"` at line 113 runs _after_ the directory discovery loop. If the loop found a `tests/` directory, it would have already set `TEST_PATH`, and the quick mode override would silently overwrite it. This only matters if `--quick` is used with a module path that happens to have a `tests/` subdirectory — unlikely but surprising behavior.

**Fix:** Move the quick mode check before the discovery loop, or skip the loop entirely when `MODE == "quick"`.

---

### C-5: Environment variable size limits for large codebases

**File:** `test-review-request.sh:202-205`
**Severity:** IMPORTANT

**Issue:** Source and test content are passed to Python via environment variables (`export _SOURCE_CONTENT="$STANDARDS_CONTENT"` etc.). On many systems, the total environment size is capped (commonly ~128KB on Linux, ~256KB on macOS). For large modules with many source files, this will silently truncate or fail with "Argument list too long." The specialist-review.sh avoids this by using file-based substitution with Python reading files directly.

**Fix:** Pass content via temporary files rather than environment variables, consistent with how `specialist-review.sh` handles its diff content.

---

### C-6: Full prompt passed as CLI argument hits ARG_MAX

**File:** `test-review-request.sh:267`
**Severity:** IMPORTANT

**Issue:** The `$SYSTEM_PROMPT` variable contains the full inlined content (source code, tests, standards, workflow docs) — potentially tens of thousands of characters. This is passed as a command-line argument to `claude`. Command-line argument length is limited by `ARG_MAX` (typically ~256KB on macOS, ~2MB on Linux). This will fail for large modules. Additionally, the full source code becomes visible in process listings (`ps aux`).

**Fix:** Write the system prompt to a file and use `--append-system-prompt` with a file path if the CLI supports it, or use a different flag/approach to pass the prompt. Check `specialist-review.sh` for comparison — it passes the full prompt as the _user message_ argument, but the same size concern applies there too.

## Security Review

### S-1: `--dangerously-skip-permissions` bypasses all safety checks

**File:** `test-review-request.sh:264`
**Severity:** IMPORTANT

**Issue:** The Claude CLI is invoked with `--dangerously-skip-permissions`, which disables all permission prompts. Combined with the Write tool being allowed, the spawned Claude process can write to any location on the filesystem without user confirmation. While the audit prompt instructs Claude to write only to the specified output file, a prompt injection via malicious source code content (inlined into the system prompt) could instruct Claude to write elsewhere. The content of arbitrary source files is inlined directly into the prompt without sanitization.

**Suggested fix:** Consider restricting the `--tools` allowlist more tightly if possible, or validate that the output file was created in the expected directory after invocation. Document the trust boundary — this script assumes source code is trusted content.

---

### S-2: Source code inlined into prompt without sanitization — template injection risk

**File:** `test-review-request.sh:130-150` (read_file_or_dir function), used at lines 154, 160
**Severity:** MINOR

**Issue:** All source and test files are `cat`-ed and inlined directly into the system prompt. If a source file contains adversarial content (e.g., markdown that looks like prompt instructions, or template injection strings like `{{OUTPUT_FILE}}`), it could interfere with the prompt template substitution. The Python template engine uses simple `str.replace()`, meaning a source file containing `{{TESTING_STANDARDS}}` would be replaced with the actual standards content, causing duplication and confusion.

**Suggested fix:** Perform template substitution in a specific order: replace standards/workflow first, then source/test content last (which is already the case in the Python script at lines 215-222). However, if source content itself contains `{{TEST_CONTENT}}`, the later replacement would substitute within the source. Consider using a substitution that can't collide (e.g., unique delimiters with UUIDs, or a proper templating library).

---

### S-3: Temp file in world-readable `/tmp` directory

**File:** `test-review-request.sh:199`
**Severity:** MINOR

**Issue:** `mktemp /tmp/test-review-request-system.XXXXXX` creates a temporary file in `/tmp` with predictable prefix. While `mktemp` generates a random suffix, the file contains the full system prompt (including source code). Other users on a shared system can read this file between creation and deletion.

**Suggested fix:** Use `mktemp` without a path (uses `$TMPDIR` which may be per-user on macOS) or set permissions: `umask 077` before `mktemp`, or `chmod 600` the file immediately after creation.

**File:** `SKILL.md:71-99` vs `test-review-request.sh:1-21`
**Severity:** NIT

**Issue:** The usage examples in SKILL.md and the script header comment document the same interface. They could drift apart over time.

### M-3: `read_file_or_dir` builds large strings via concatenation in a loop

**File:** `test-review-request.sh:124-151`
**Severity:** MINOR

**Issue:** The function accumulates content into a bash variable via repeated concatenation. For directories with many files, this is O(n^2) string concatenation in bash. More importantly, the entire content is held in memory as a single bash variable, then exported as an environment variable — hitting the limits noted in C-5.

**Suggested fix:** Write content directly to a temp file and read it back, rather than accumulating in a variable.

---

### M-4: Inconsistent timeout defaults between the two scripts

**File:** `specialist-review.sh:122` vs `test-review-request.sh:242`
**Severity:** NIT

**Issue:** `specialist-review.sh` defaults to `CLAUDE_TIMEOUT=300` (5 min) and `CLAUDE_MAX_TURNS=25`, while `test-review-request.sh` defaults to `CLAUDE_TIMEOUT=120` (2 min) and `CLAUDE_MAX_TURNS=3`. The test audit inlines all content so it needs fewer turns, which explains the difference. But the 120s timeout may be tight for large modules where Claude needs to analyze substantial inlined content.

**Suggested fix:** Document why the defaults differ, or increase the test-review timeout default to match.

### A-4: Two different content-passing strategies across the two scripts

**File:** `specialist-review.sh:97-116` vs `test-review-request.sh:194-226`
**Severity:** MINOR

**Issue:** `specialist-review.sh` passes the diff as part of the user message (via `$PROMPT`) and uses `sed` + Python for template substitution. `test-review-request.sh` passes all content via `--append-system-prompt` and uses only Python with environment variables for substitution. **Suggested fix:** Establish a common pattern. The ideal approach would be: write the full prompt to a temp file, pass it via a file-path flag (or pipe), and let the CLI read from disk instead of the command line.

## Phase 3: Synthesis

### Summary

This is a well-structured consolidation of two related skills into a single, comprehensive agent workflow document. The SKILL.md is thorough and provides clear separation of responsibilities between implementer agents and Claude reviewers. The main technical risks are in the `test-review-request.sh` script, where content passing via environment variables and command-line arguments will break on large codebases.

### Findings by Severity

**Critical:** None

**Important:**
- C-1: `--tests`/`--output` argument parsing crashes silently when flag is last argument
- C-5: Environment variable size limits will fail on large codebases
- C-6: Full prompt passed as CLI argument hits ARG_MAX on large modules
- S-1: `--dangerously-skip-permissions` with untrusted inlined source code creates prompt injection surface

**Minor:**
- C-2: `--quick` error message doesn't hint at expected positional argument
- C-4: Quick mode silently overrides discovered test path
- S-2: Template substitution collisions possible with adversarial source content
- S-3: Temp file in `/tmp` world-readable on shared systems
- M-1: 521-line SKILL.md is dense (but well-organized)
- M-3: O(n^2) string concatenation in `read_file_or_dir`
- A-3: Unused `REPO_ROOT` adds unnecessary git dependency
- A-4: Inconsistent content-passing strategies between the two scripts

**Nits:**
- C-3: `REPO_ROOT` computed but never used (dead code)
- M-2: Duplicated invocation docs between SKILL.md and script headers
- M-4: Inconsistent timeout defaults between scripts (120s vs 300s)

### Cross-Cutting Concerns

**Correctness + Security (C-5/C-6 + S-1/S-2):** The content-passing mechanism is both a correctness problem (env vars and CLI args hit size limits) and a security concern (source code exposed in process listings and prompt injection via inlined content). These are the same root cause — large amounts of untrusted content flowing through channels not designed for it. Fixing the correctness issue (writing to temp files instead) would simultaneously reduce the security surface.

**Correctness + Architecture (C-5/C-6 + A-4):** The two scripts use different content-passing strategies, and the newer script (`test-review-request.sh`) chose the less scalable approach (env vars → Python → CLI args). Converging on the approach used by `specialist-review.sh` (file-based) would fix the scalability issues.

**Maintainability + Correctness (M-3 + C-5):** The `read_file_or_dir` function's in-memory string accumulation feeds directly into the env var export. Fixing M-3 (write to temp file) would naturally lead to fixing C-5 (pass file path instead of content).

### Risk Interactions

**C-5 + C-6 compound to create a hard ceiling:** The env var limit (~256KB on macOS) AND the CLI arg limit (~256KB) are hit in sequence by the same content. A module with just 50-100 source files of moderate size will exceed both limits, making the test audit script unusable for real-world projects of non-trivial size. The specialist-review script doesn't have this problem because diffs are typically smaller than full source trees.

**S-1 + S-2 amplify each other:** The prompt injection risk from inlined source (S-2) is amplified by `--dangerously-skip-permissions` (S-1). If source content can influence the prompt, and the spawned Claude has unrestricted filesystem write access, the combined risk is that a malicious source file could cause Claude to write arbitrary files anywhere. In practice the threat model assumes trusted source code, but this should be explicitly documented.

### Verdict

**APPROVE WITH CHANGES**

The consolidation is architecturally sound and the documentation quality is excellent. The two IMPORTANT correctness findings (C-5, C-6) represent a real scalability wall that will be hit on non-trivial modules and should be addressed before this becomes the primary test audit path. The argument parsing edge case (C-1) is a minor usability issue. Everything else is improvement territory.
