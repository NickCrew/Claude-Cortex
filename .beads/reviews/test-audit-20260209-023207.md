## Notes

### Test File Mismatch

The specified test file (`tests/unit/core/test_skills.py`) tests a completely different module (`claude_ctx_py.core.skills`). A new test file should be created specifically for `scripts/validate_registry.py`, likely at `tests/unit/scripts/test_validate_registry.py`.

### Test Infrastructure Needs

1. **Fixtures for registry data**: Tests will need helper fixtures that create minimal valid/invalid YAML registries, JSON schemas, and author files in temp directories. Consider a `tmp_path`-based fixture that scaffolds the expected file layout.

2. **Path mocking**: The module uses `PROJECT_ROOT = Path(__file__).parent.parent` to find files. Tests will need to either monkeypatch these module-level constants or use temp directories that mirror the expected layout.

3. **stdout capture**: Several methods print directly to stdout. Tests should use `capsys` to verify output messages without cluttering test output.

4. **Schema fixture**: A minimal valid JSON schema is needed for `validate_schema()` tests. This could be a simplified version of the real `registry.schema.json` or a purpose-built test schema.

### Dependency Graph Testing

The cycle detection algorithm (`has_cycle`) uses DFS with a recursion stack. This is a classic algorithm with known edge cases that should be tested:
- Self-referencing dependency (A→A)
- Two-node cycle (A→B→A)
- Long cycle (A→B→C→D→A)
- Diamond dependency (A→B, A→C, B→D, C→D) — no cycle
- Multiple disconnected components, one with a cycle
- Empty dependency graph (no skills)

### `validate_statistics` has a latent bug risk

The `status_counts` dict is initialized with only four keys: `active`, `maintenance`, `deprecated`, `archived`. If a skill has any other status value (e.g., `"beta"`, `"draft"`), the `.get(status, 0) + 1` fallback will create a new key but the count won't be validated against anything. This won't crash but could silently hide miscounts.
