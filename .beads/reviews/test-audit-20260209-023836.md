## 5. Bug Notes

### Potential Bug: Cycle Detection with Shared `visited` Set

In `validate_dependencies`, the cycle detection loop:

```python
visited: set[str] = set()
for skill_id in skills:
    if skill_id not in visited:
        if has_cycle(skill_id, visited, set()):  # <-- fresh rec_stack each iteration
            valid = False
```

Each top-level iteration creates a **fresh `rec_stack`** but shares `visited`. This means if the first DFS from node A visits nodes B and C, a subsequent top-level call will skip B and C entirely (they're in `visited`). This is correct for standard DFS cycle detection in directed graphs — but only if the `rec_stack` is also shared or the algorithm is implemented differently. With a fresh `rec_stack` per top-level call, certain cross-component back edges could be missed.

**Example that may be missed:**
- Skills: A->B, C->B->C (cycle in component {B,C})
- First DFS from A visits A, B. `visited = {A, B}`.
- Next, DFS from C: visits C, then B is already in `visited` but NOT in `rec_stack` (fresh set). So the `elif dep in rec_stack` check fails, and the cycle C->B->C is **not detected**.

This should be confirmed with a test and fixed if confirmed.
